# First Issues Guide

This document explains how beginner-friendly issues should be identified and what contributors should expect from them.

## Labels To Look For

- `good first issue`
- `help wanted`
- `docs-help`
- `design-help`
- `testing-help`

## What A Good First Issue Should Include

- one clear problem
- one clear goal
- acceptance criteria
- lane label
- enough context to find the relevant files

## What A Good First Issue Should Not Be

- broad architecture change
- auth, infra, or release-critical work
- cross-lane rewrite
- issue with unclear ownership

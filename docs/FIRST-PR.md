# First PR Guide

This document is for first-time contributors.

## Best First PR Characteristics

Your first PR should be:

- small
- issue-linked
- easy to review
- limited to one lane
- low-risk

## Good First PR Examples

- docs clarification
- issue template improvement
- label/doc consistency fix
- isolated UI cleanup in one surface
- test improvement for an existing behavior

## Avoid For First PR

- auth rewrites
- infra changes
- contract changes
- release workflow edits
- broad refactors
- mixed-scope PRs

## Before Opening the PR

- sync from latest `main`
- rerun the relevant checks
- confirm your branch still matches the issue scope
- mention any assumptions or blockers clearly
